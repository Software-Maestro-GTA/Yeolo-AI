# Progress Manager Skill (단일 기능 진척도 관리 지침)

이 문서는 하나의 백로그 기능 개발 태스크를 시작할 때 생성하는 `progress.md`의 관리 가이드라인입니다. 각 에이전트는 기동 및 작업 완료 시점에 맞춰 본 스킬 규격에 따라 `progress.md` 파일을 성실히 업데이트해야 합니다.

---

## 에이전트별 progress.md 갱신 규칙 (Operational Rules)

### 1. Planner 단계 (기획 및 체크리스트 구성)
* **목적**: 기획 요구사항을 코드로 만들기 위한 상세 구현 가이드를 수립합니다.
* **동작 규칙**:
  * [templates/progress_template.md](../../templates/progress_template.md)를 복사하여 프로젝트 내에 `progress.md` 파일을 생성합니다.
  * 백로그 기본 정보를 명세하고, `specs/` 내 사양서들을 분석하여 소스코드에 반영할 **세부 기획 및 구현 체크리스트(`- [ ]`)**를 구체적으로 쪼개어 채워 넣습니다.
  * 본인의 `Agent Execution Log`의 진행 상태를 `[완료]`로 바꾸고 분석 요약 내용을 작성합니다.

### 2. Tester 단계 (테스트 이력 기록)
* **목적**: 테스트 시나리오 설계 및 테스트 코드 구현 상태 전이를 이력으로 남깁니다.
* **동작 규칙**:
  * 작업을 시작하는 즉시 `Agent Execution Log`의 `Tester` 구역의 진행 상태를 `[진행중]`으로 표기합니다.
  * 테스트 코드 작성이 완료되면 진행 상태를 `[완료]`로 변경하고, 생성/수정한 테스트 파일 경로(예: `tests/store.test.ts`)와 검증 방식을 상세히 기재합니다.

### 3. Coder 단계 (구현 이력 기록)
* **목적**: 프로덕션 비즈니스 코드 및 UI 컴포넌트의 구현 이력을 남기고 체크리스트 진척도를 업데이트합니다.
* **동작 규칙**:
  * 작업을 시작하는 즉시 `Agent Execution Log`의 `Coder` 구역의 진행 상태를 `[진행중]`으로 표기합니다.
  * 코드 구현이 완료되면 진행 상태를 `[완료]`로 변경하고, 구체적으로 생성/수정한 파일 경로 및 로컬 테스트 최종 성공 내역을 상세히 기재합니다.
  * Coder는 자신이 구현을 완료할 때마다 `세부 기획 및 구현 체크리스트`에서 자신이 완료한 항목의 마크를 `[x]`로 업데이트합니다.

### 4. Reviewer 단계 (검증 승인 및 완료 마킹)
* **목적**: 코드 리뷰 및 빌드 무결성 검증 통과를 인증하고 파이프라인을 최종 종료합니다.
* **동작 규칙**:
  * 모든 검증 도구(Lint, 컴파일러, Jest 등)가 최종 통과하면, 진행 상태를 `[완료]`로 바꾸고 검증 세부 내역(LINT/TYPE/TEST의 PASS 상태)을 수록합니다.
  * 1번(Planner)부터 4번(Reviewer 자신)까지의 모든 에이전트 상태가 `[완료]`인지 최종 검토 후, `progress.md` 파일 자체의 상태를 완료(DONE)로 마킹하고 전체 파이프라인 작업을 끝마칩니다.
