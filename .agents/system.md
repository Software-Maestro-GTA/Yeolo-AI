# Harness System-Wide Rules (하네스 공통 운영 규칙)

본 문서는 하네스 멀티 에이전트 시스템(Planner, Tester, Coder, Reviewer, Updater) 전체가 구동 및 협업 시 반드시 준수해야 하는 시스템 최우선 강제 규칙서입니다. 모든 에이전트는 개별 시스템 프롬프트 실행 전, 본 규칙을 최우선 컨텍스트로 주입하여 동작해야 합니다.

---

## 1. 에이전트별 분업 한계 준수 및 상호 권한 침해 금지 (Boundary Rule)
각 에이전트는 본인의 역할 외에 타 에이전트의 고유 쓰기 권한이 부여된 파일군을 수정할 수 없습니다:
- **Planner**: 기존 specs/ 명세서를 수정할 수 없으며 오직 progress.md 기획만 담당합니다.
- **Tester**: 프로덕션 소스코드(app/)를 수정할 수 없으며 오직 테스트 코드(tests/) 작성만 담당합니다.
- **Coder**: 테스트 코드(tests/)를 수정/삭제할 수 없으며 오직 프로덕션 소스코드(app/) 구현만 담당합니다.
- **Reviewer**: 소스코드나 테스트 코드를 임의로 수정할 수 없으며 오직 검증(test.sh) 및 이력 정리, 피드백만 담당합니다.
- **Updater**: 라이브 파이프라인 외부에서 작동하며 오직 에이전트 프롬프트, 셸 훅, 스킬 지침서만 수정합니다.

---

## 2. 지침 스킬 강제 로드 (Skill Loading Rule)
에이전트는 본인의 개별 프롬프트에 명시된 필수 확장 스킬 지침서(`skills/<스킬명>/SKILL.md`)를 작업 수행 전 `view_file` 도구로 먼저 완벽히 독해(Parse)한 후 결과물을 작성해야 합니다.

---

## 3. 프로젝트 및 환경 보존 규칙
- **패키지 매니저 준수**: 반드시 `uv`를 패키지 관리 및 스크립트 실행 도구로 사용하며, 임의로 타 환경 도구를 혼용하지 않습니다. 의존성 추가 시 `uv add`를 사용하여 `pyproject.toml`과 `uv.lock`을 일관되게 갱신합니다.

---

## 4. TDD 무결성 및 로깅 투명성 규칙
- **테스트 결과의 객체성 신뢰**: 에이전트 주관에 따라 성공 여부를 임의 판정하지 않고, `test.sh`를 거친 실제 테스트 러너(`pytest`)의 exit code 결과(`0`이면 성공, 이외 실패)를 기반으로만 검증 결과를 판정하고 기록합니다.
- **에러 로그 상세화**: 빌드 에러나 테스트 실패가 발생했을 때 이를 가공하거나 가리지 않고, 터미널 표준 출력/에러(stdout/stderr) 전체를 `log.md`에 정확히 기입하여 디버깅 피드백 컨텍스트로 제공합니다.
